# Заполнение трёх real-case templates

Эта инструкция повторяется независимо для `REAL-CASE-01`, `REAL-CASE-02` и
`REAL-CASE-03`. Канонический [`template_manifest.json`](template_manifest.json)
намеренно неисполняем: в нём стоят placeholder-пути и нулевые SHA-256, reviewers
не указаны, а `signoff_status` равен `unsigned`. В этом PR эти три шаблона не
получают ни статус `approved`, ни статус `accepted`.

Создайте контролируемую копию нужной записи в отдельном manifest под каталогом
`acceptance/`. Не исправляйте исходный raw и не превращайте канонический template
в свидетельство приёмки. Пути разрешаются относительно файла manifest: текущие
placeholder-пути `real_cases/...` рассчитаны на manifest, расположенный прямо в
`acceptance/`, а не на запуск `template_manifest.json` из его собственного каталога.

## Назначение шаблонов

| Template | Источник | Дополнительный gate |
|---|---|---|
| `REAL-CASE-01-PENDING` | подписанный лабораторный CSV | обычная независимая проверка real case |
| `REAL-CASE-02-PENDING` | исходный XLSX, импортированный в production strict mode | сохранить точные байты workbook и доказательство выбранного листа/заголовка |
| `REAL-CASE-03-PENDING` | подписанный протокол, перенесённый через ручной ввод | отдельная инженерная проверка эквивалентности raw, черновика и audit trail |

## 1. Исходный файл

1. Получите подписанный лабораторный протокол и сохраните его точные байты в
   стабильном каталоге evidence. Не очищайте, не пересохраняйте, не исправляйте
   форматирование и не заменяйте исходник экспортом программы.
2. Установите фактический `source_type` (`csv` или `xlsx`) и замените
   `input_files.protocol.path` на безопасный относительный путь.
3. Рассчитайте SHA-256 именно сохранённых байтов и замените нулевой
   `input_files.protocol.sha256`. Хэш отчёта или подготовленной таблицы не является
   хэшем raw.
4. Для ручного ввода сохраните отдельно подписанный источник, lossless manual draft
   и audit/equivalence evidence. Gate `manual_input_engineering_review` остаётся
   `unsigned`, пока инженер не сопоставит их независимо.

## 2. Metadata

Создайте production metadata JSON только из подписанных фактов, задайте
`data_class=real` и замените путь и нулевой SHA-256 в `input_files.metadata`.
Metadata должны явно фиксировать:

- идентификатор и дату испытания, единицы, форму, диаметр и площадь штампа;
- исходную последовательность ступеней, ветви и статусы событий;
- назначения каналов, паспорта, поверку на дату опыта, координаты индикаторов и
  неизменную политику агрегации;
- фактическое решение о failure/right-censoring без выдуманной осадки;
- профиль `E`, выбранный диапазон давления, автора решения, UTC-время, причину и
  статус рассмотрения;
- реальные `pair_id`, группы и явное решение о публикационной кривой;
- provenance и SHA-256 всех исходных свидетельств.

Пока хотя бы одно поле не подтверждено, сохраняйте `template_only=true`,
`status=pending_real_data`, пустой reviewer и `signoff_status=unsigned`.

## 3. Независимый расчёт

Подготовьте отдельный расчёт, который не копирует значения из Soil Stamp. Он должен
содержать автора, дату и версию документа, хэши источников, формулы, исходные
величины и единицы, промежуточные действия, ожидаемые численные результаты,
решения по failure/censoring и `E`, а также однозначные поля для сопоставления.

В `independent_calculation_reference` укажите устойчивый путь или идентификатор
контролируемого расчёта. Рекомендуется также добавить документ как отдельный
`input_files`-объект с `path` и `sha256`. Значение `null` не обеспечивает
методическую трассируемость real case.

## 4. Reviewer

Точные хэши raw/metadata, независимый расчёт и отчёты программы проверяет реальный
ответственный инженер, а не разработчик или автоматический агент. Только этот
инженер заполняет:

- `reviewer.name`;
- `reviewer.organization`;
- `reviewer.reviewed_at` в ISO 8601 UTC.

Reviewer не заполняется заранее и не подменяется автором программы.

## 5. Допуски

До просмотра расхождений задайте для каждого численного поля абсолютный и/или
относительный допуск, его единицу, источник и инженерное обоснование. Свяжите
golden value с именованным допуском из `tolerances`. Строковые статусы, flags,
предупреждения и хэши проверяются строго.

Не выводите допуски из результата программы и не расширяйте их задним числом.
Заполните `expected_outputs`, `golden_values`, `expected_warnings` и
`expected_review_status` только по независимому свидетельству.

## 6. Подпись

Инженерная подпись должна однозначно связывать:

- `case_id`;
- SHA-256 raw, metadata и независимого расчёта;
- заранее объявленные допуски;
- SHA-256 проверенных output/report artifacts;
- решение, имя подписанта и UTC-время.

Храните подписанное свидетельство неизменяемо. Рекомендуемый вариант — добавить
`input_files.signoff` с безопасным относительным `path` и точным `sha256`, чтобы
acceptance-run проверял его байты. Технический PASS или зелёный CI подписью не
является.

## 7. `signoff_status`

До завершения всех шести предыдущих шагов оставляйте строго
`signoff_status=unsigned`; провал engineering-signoff check в этом состоянии
является ожидаемым и правильным. Только после реальной подписи и заполнения reviewer
ответственный инженер может установить `signoff_status=signed` в отдельно
рассмотренной исполняемой копии и снять `template_only`.

Не используйте для real cases значения `approved` или `accepted`. Не меняйте три
канонических шаблона, чтобы имитировать завершение. После подписи запустите отдельный
reviewed manifest, сохраните JSON/Markdown/HTML acceptance reports и независимо
проверьте все input/report hashes.

## Контроль перед передачей инженеру

- [ ] исходные bytes и SHA-256 сохранены без исправлений;
- [ ] metadata отражают подписанные факты и решения;
- [ ] независимый расчёт существует и не скопирован из программы;
- [ ] допуски объявлены до сравнения и имеют единицы/обоснование;
- [ ] reviewer является ответственным инженером;
- [ ] подписанное свидетельство связано с точными хэшами;
- [ ] неподписанный case сохраняет `signoff_status=unsigned`;
- [ ] успешный synthetic/CI run не трактуется как приёмка реального испытания.
