# Soil Stamp Antonov 0.5.0b2.dev1

[![CI](https://github.com/an89537171644/geo_graf/actions/workflows/ci.yml/badge.svg)](https://github.com/an89537171644/geo_graf/actions/workflows/ci.yml)

Локальная программа для воспроизводимой обработки штамповых испытаний армированного и неармированного грунта. Она строит кривые «нагрузка/давление — осадка» в ориентации В. М. Антонова, сохраняет исходную последовательность протокола, рассчитывает `pcr`, условный штамповый модуль, групповые доверительные интервалы и сравнительные показатели. Версия `0.5.0b2.dev1` добавляет ручной ввод первичных данных поверх того же проверенного pipeline и не меняет научную логику графиков Антонова или модель разрушения.

## Быстрый запуск в Windows

1. Установите Python 3.10 или новее.
2. Дважды щёлкните `run_app.bat` либо выполните в PowerShell:

```powershell
./run_app.ps1
```

При первом запуске рядом с программой будет создано изолированное окружение `.venv` и установлены зависимости. Затем откроется локальный интерфейс Streamlit. По умолчанию загружается демонстрационная серия.

Ручной запуск:

```powershell
py -3 -m venv .venv
./.venv/Scripts/python -m pip install -r requirements.txt
./.venv/Scripts/python -m streamlit run app.py
```

## Ручной ввод (TASK 12)

1. В боковой панели выберите источник **«Ввод вручную»**.
2. Заполните паспорт опыта и внесите исходные показания в таблицу. Числовые raw-значения сохраняются текстом, включая десятичную запятую и завершающие нули.
3. Исправьте адресные блокирующие ошибки. Предупреждения, переходы шкалы, обратный ход и коррекции остаются видимыми и попадают в журнал.
4. Сохраните черновик JSON либо откройте ранее сохранённый файл. Критическая ошибка блокирует анализ, но не сохранение черновика.
5. Нажмите **«Передать snapshot в анализ»**. Только эта явная операция замораживает текущую ревизию и передаёт её в общий расчётный pipeline.

Поддерживаются 1–4 индикатора, ветви `loading`, `hold`, `unloading`, `reloading`, `cyclic` и статусы `measurement`, `failure`, `instrument_limit`, `stopped_without_failure`, `invalid`. Строка `failure` может не иметь показания индикатора: разрушение фиксируется как событие без выдуманной осадки. Ручной и Excel-ввод используют один `prepare_measurements()`; отдельной научной формулы у GUI нет.

Черновик схемы `manual-entry-draft/1.2` хранит паспорт, raw-строки, порядок, UUID строк, автора, временные метки и append-only audit. Поля `baseline_group` и `pair_id` независимы: первое называет контрольную серию, второе явно связывает конкретную пару/блок опытов. Пустой `pair_id` означает отсутствие заявленной парности и никогда не заполняется из `baseline_group`. Для `indicator_1..4` и `reference_indicator` сохраняются независимые паспорта; общей метрологической записи в новом формате нет. Явная команда GUI может скопировать уже заполненный паспорт в выбранные каналы, но создаёт независимые копии и записывает причину в audit.

Черновики `1.0` и `1.1` открываются последовательной lossless-миграцией. Старый общий паспорт сохраняется только как read-only `legacy_common_indicator_passport`, получает `metrology_status=migration_review_required` и не участвует в расчёте. Инженер вручную назначает первый поканальный паспорт, при необходимости явно копирует его и отдельно подтверждает метрологию. Машиночитаемый контракт: `docs/manual-entry-draft-1.2.schema.json`; пример: `examples/manual_entry_demo.json`. SQLite-архив, утверждённые ревизии, поиск и backup относятся к TASK 13–15 и в этой версии намеренно недоступны.

`pair_id` сохраняется буквально: пробелы по краям не удаляются скрыто, а вызывают видимое предупреждение и запрещают использовать такое значение как подтверждение парности.

## Формат протокола

CSV и строгий XLSX приводятся к одной канонической long-table. Исходный Excel открывается только для чтения и не изменяется. Для каждой импортированной точки сохраняются `sheet_name`, `source_row`, `source_columns`, `sequence_index`, `raw_stage`, `raw_load`, `raw_indicator`, `parsed_stage`, `parsed_load` и `parsed_indicator`.

Минимальные обязательные столбцы:

```text
test_id,stage,load
```

Должен присутствовать `settlement` либо хотя бы один из `indicator_1..4`. Полная поддерживаемая схема:

```text
test_id,stage,branch,timestamp,load,
indicator_1,indicator_2,indicator_3,indicator_4,
reference_indicator,horizontal_indicator,settlement,
status,comment,group
```

- смысл `load` задаётся полем `load_kind`: `force` для силы или `pressure` для уже рассчитанного давления;
- для `load_kind: "force"` поддерживаются `N`, `kN`, `MN`, `kgf`, `tf`; сила умножается на `load_factor` и `lever_ratio`, а `p = F / A`, поэтому для давления нужна `stamp_area_m2` либо круглый `stamp_diameter_mm`;
- для `load_kind: "pressure"` поддерживаются `Pa`, `kPa`, `MPa`; построение `p–s` не требует площади, но силовые показатели и оси остаются недоступными без геометрии штампа;
- `settlement`, если он заполнен, имеет приоритет и никогда не подменяется показаниями индикаторов;
- если `settlement` отсутствует, расчёт из `indicator_1..4` разрешён только после проверки поканальных `indicator_passports`, подтверждения их назначения и выбора явной политики `settlement_aggregation`; старый общий паспорт сохраняется для миграционной проверки, но не распределяется по каналам автоматически;
- без подтверждённой калибровки `indicator_*` остаются raw: расчёт осадки блокируется, а при наличии прямой `settlement` индикаторные функции центра/крена отключаются;
- режимы исходной шкалы: `increasing`, `increasing_wrapped`, `decreasing`, `decreasing_wrapped`; готовая накопленная осадка: `cumulative_settlement`;
- автоматический переход через ноль применяется только при единственном физически допустимом номере оборота; неоднозначный переход блокируется до явного `{channel}_turn_number`;
- пропуски не заполняются, небольшой обратный ход не обнуляется, коррекция нуля применяется только из паспорта; все такие события записываются в `indicator_processing_events.csv`;
- порядок строк является порядком измерения и не сортируется по нагрузке;
- машинные CSV/JSON используют десятичную точку, локализация применяется только к человекочитаемому отчёту.

Пример находится в `examples/demo_protocol.csv`.

### Режимы Excel-импорта

- `strict` — режим по умолчанию для нового проекта; неизвестный или неоднозначный заголовок блокирует расчёт и указывает лист, строку и столбец;
- `interactive` — использует явно подтверждённое сопоставление канонических полей с заголовками/буквами Excel; mapping можно скачать из интерфейса и повторно применить в CLI;
- `heuristic` — только для совместимости с legacy-блоками вида `N испытание`; всегда создаёт предупреждение и не скрывает необходимость проверки шкалы индикатора.

В `strict`/`heuristic` интерфейс по умолчанию импортирует все листы, чтобы проверять межлистовые дубли `test_id` и смешанные схемы. Ограничение одним листом выбирается явно и фиксируется warning; в `interactive` один лист является частью подтверждённого mapping-scope.

Неизвестные столбцы не заменяются соседними, пустой номер ступени не придумывается, а фактическая строка разрушения без показания осадки не получает фиктивное значение.

Перед открытием книга проверяется как недоверенный ZIP/XML: ограничиваются размер файла, число entries, распакованный объём, коэффициент сжатия, количество листов и заявленные размеры листов. XLSM читается только как данные; VBA не загружается и не выполняется.

В `tests` можно переопределить геометрию, группу, параметры грунта, армирование и отдельные поля того же поканального паспорта для испытания; записи разных каналов не объединяются между собой. Непустой `tests` одновременно считается реестром опытов проекта: в strict/interactive он должен содержать каждый `test_id` выбранного протокола; лишние записи создают warning. Если индивидуальных overrides нет, поле можно не задавать. `baseline_group` — необязательное имя контрольной серии, а `pair_id` — необязательный ID конкретной пары/блока. Парный анализ разрешён только для полного взаимно-однозначного набора одинаковых непустых `pair_id` в обеих группах; иначе применяется независимый анализ с видимым предупреждением. Полный обычный пример — `examples/demo_metadata.json`, пример четырёх переходов шкалы — `examples/indicator_calibration_demo.csv` и `examples/indicator_calibration_demo.json`.

`settlement_aggregation` обязателен для формирования осадки из индикаторов. `all_channels_mean` блокирует строку при пропуске любого заранее активного канала; `selected_channels_mean` использует неизменяемый список; `primary_channel` — один заранее выбранный канал; `plane_center` требует минимум три неколлинеарных поканальных координаты и сохраняет rank, residual RMS и крен; `no_aggregation` осадку не формирует. Для каждой строки экспортируются `channels_required`, `channels_used`, `missing_channels` и `aggregation_status`, поэтому программа не меняет знаменатель среднего молча. Срок поверки оценивается только относительно `experiment_date`; при неизвестной дате сохраняется `review_required`, а системная дата не входит в научный результат.

Публичный вызов `prepare_measurements()` по умолчанию также проверяет явность критических физических metadata. Режим совместимости со старыми программными вызовами включается только явно: `strict_metadata=False`; интерфейс и CLI применяют его лишь после отдельной строгой проверки проекта.

### Методический контракт `E_stamp_app`

Параметры условного штампового модуля задаются вложенным объектом `modulus_method`. Точный контракт (токены `P_MIN_KPA` и `P_MAX_KPA` нужно заменить числами, они не являются диапазоном по умолчанию):

```json
{
  "modulus_method": {
    "profile_id": "antonov_round_stamp_v1",
    "nu": 0.30,
    "shape_factor": 0.80,
    "p_range_kPa": ["P_MIN_KPA", "P_MAX_KPA"],
    "p_range_source": "explicit",
    "approval": {
      "status": "approved",
      "author": "Фамилия И.О.",
      "timestamp_utc": "YYYY-MM-DDThh:mm:ssZ",
      "reason": "Обоснование выбора диапазона и метода"
    }
  }
}
```

`p_range_source` принимает `explicit`, `accepted_pcr` или `project_profile`. Для `accepted_pcr` нужно ручное подтверждение в текущем `PCRResult` либо сохранённая запись внутри `modulus_method`:

```json
"accepted_pcr": {
  "value_kPa": "PCR_KPA",
  "accepted_by": "Фамилия И.О.",
  "accepted_at": "YYYY-MM-DDThh:mm:ssZ",
  "reason": "Обоснование принятия pcr"
}
```

Автоматически оценённый `pcr_auto` не считается принятым и не может молча задать диапазон. Тот же `modulus_method` можно задать в `tests.<test_id>.modulus_method`; его непустые ключи заменяют глобальные. Приоритет источников от младшего к старшему: глобальные metadata → `tests.<test_id>` → ручное подтверждение GUI → CLI overrides. Источник `nu`, `shape_factor`, диапазона и профиля сохраняется с каждым результатом.

Доступны три версионированных профиля:

| Профиль | Контракт и статус |
| --- | --- |
| `antonov_round_stamp_v1` | Круглый штамп, `nu=0.30`, `shape_factor=0.80`, требует явного диапазона. Это проектный контракт: библиографическая ссылка в репозитории пока не зафиксирована; профиль не является нормативным. |
| `custom_v1` | Проектные `nu`, `shape_factor` и диапазон; без полного контракта и утверждения требует проверки. |
| `diagnostic_unapproved_v1` | Явно неутверждённый legacy/диагностический режим; его результат не становится основным. |

Условный штамповый модуль — расчётный результат по формуле и выбранному участку кривой. Нормативный модуль требует отдельной ссылки на действующую нормативную методику и подтверждённую область её применимости; такого утверждённого нормативного профиля в текущей версии нет. Утверждённый методический профиль означает лишь версируемый и проверяемый контракт расчёта внутри проекта и сам по себе не делает модуль нормативным.

Без подтверждённого диапазона программа не выбирает «подходящий» участок молча. Она может вернуть числовой диагностический результат по наблюдённому диапазону, но он явно помечается `is_primary=false`, `review_status=review_required` и `p_range_source=diagnostic_full_curve`. Именно так обрабатывается существующий demo, в metadata которого диапазон не задан. Для человекочитаемого вывода `E_stamp_app` печатается в МПа с округлением по оценённой неопределённости.

## Научные правила и безопасные допущения

1. `pcr` рассчитывается только по устойчивым измеренным точкам первой ветви `loading`, минимум по три различных уровня давления с каждой стороны излома. Bootstrap одиночной кривой — wild residual с распределением Радемахера; seed сохраняется.
2. Групповой bootstrap выполняется по целым испытаниям. Интерполяция кусочно-линейная и только внутри наблюдаемого диапазона; экстраполяции нет.
3. Формула условного модуля в текущей конфигурации:

   `E_stamp_app = (1−ν²) · K_shape · D · dp/ds`.

   `ν`, `K_shape`, диапазон давления и метод всегда записываются вместе с результатом. До фиксации нормативной методики проекта результат не следует называть нормативным модулем деформации.
4. `seating_corrected` принимает только явное смещение для каждого испытания. Программа намеренно не придумывает автоматический алгоритм посадочной поправки.
5. Выбросы не удаляются автоматически. Исключение наблюдений должно оформляться отдельным проверяемым решением; текущая версия сохраняет все исходные строки.
6. Остаточная осадка определяется при реально измеренном минимальном давлении разгрузки, без автоматической экстраполяции к нулю.
7. Осадка центра и направление крена рассчитываются только при подтверждённой калибровке и координатах минимум трёх индикаторов.
8. Нормировка `p/pu` разрешается только по явно заданному `pu_kPa_confirmed`; верхняя граница ступени разрушения не подменяет точечную оценку `pu`.

## Пакетный запуск

После установки проекта (`pip install -e .`) доступна команда:

```powershell
soil-stamp examples/demo_protocol.csv examples/demo_metadata.json `
  --out results --plot-mode antonov_publication --axis p-s `
  --bootstrap 500 --seed 202604
```

Методические CLI overrides задаются явно:

```powershell
soil-stamp protocol.csv metadata.json --out results `
  --method-profile antonov_round_stamp_v1 `
  --e-range P_MIN:P_MAX --e-range-source explicit `
  --e-range-author "Фамилия И.О." --e-range-reason "Обоснование выбора"
```

`--e-range` принимает два числа в кПа через двоеточие. `--e-range-author` и `--e-range-reason` опциональны синтаксически, но без них ручно заданный диапазон не становится основанием для основного результата: `review_status` остаётся `review_required`.

Результат содержит таблицы, SVG/PDF/PNG, отчёт и `reproducibility.zip`.

Для Excel:

```powershell
soil-stamp protocol.xlsx metadata.json --import-mode strict --out results
soil-stamp protocol.xlsx metadata.json --import-mode interactive `
  --column-map excel_column_mapping.json --sheet "Протокол" --header-row 3 --out results
```

Выход дополнительно содержит `provenance.json`, `import_issues.csv`, `raw_cells.csv` для Excel и исходный файл внутри ZIP. `manifest.json` перечисляет SHA-256 каждого файла пакета.

`raw_cells.csv` является машинным provenance-файлом и намеренно сохраняет пользовательский текст/формулы без изменения. Программа их не исполняет; при просмотре в табличном редакторе импортируйте файл как текст и не разрешайте выполнение внешних формул/ссылок.

## Тесты

```powershell
./.venv/Scripts/python -m pip install -c constraints/ci.txt -e ".[test]"
./.venv/Scripts/python -m ruff check .
./.venv/Scripts/python -m compileall -q app.py soilstamp tests scripts
./.venv/Scripts/python -m pytest
./.venv/Scripts/python -m pytest tests/test_analysis.py tests/test_data.py tests/test_indicators.py tests/test_plotting.py tests/test_image_regression.py --cov=soilstamp --cov-report=term-missing --cov-fail-under=80
```

Image-regression эталон находится в `tests/baseline` и фиксирует ориентацию, сетку, рамку, маркеры и ломаные Antonov-графика.

GitHub Actions выполняет эти проверки на Windows и Ubuntu с Python 3.10, 3.11 и 3.12. Полный pytest включает Streamlit AppTest. Затем CI запускает пакетную демонстрацию, проверяет CSV/JSON/SVG/PDF/PNG и `reproducibility.zip`, публикует JUnit XML, coverage XML и демонстрационные результаты. Единый обязательный статус для правил ветки — `Required CI`.

`constraints/ci.txt` фиксирует единый набор прямых runtime/test-зависимостей, совместимый со всей матрицей Python 3.10–3.12. Платформенные транзитивные зависимости по-прежнему разрешает `pip`; отдельные полностью замороженные lock-файлы для каждой ОС и версии Python следует формировать перед выпуском установочного дистрибутива.

## Границы 0.5.0b2.dev1 / TASK 12

Ядро закрывает статическую часть 0.4, TASK 01 импорта, gate калибровки индикаторов и TASK 12 ручного ввода: raw/parsed provenance, поканальные паспорта, хэши, неизменность исходного Excel, проверяемая развёртка шкалы и lossless ручной черновик. Версия `0.5.0b2.dev1` не является общим релизом `0.5.0b2`: следующие задачи выполняются последовательно после отдельной приёмки.

В этой версии есть следующие явно обозначенные границы:

- для `E_regression` рассчитывается регрессионный доверительный интервал; поля ДИ для `E_secant`, `E_tangent` и `E_incremental_diagnostic` остаются пустыми, если метод не задаёт остаточных степеней свободы или отдельную модель неопределённости;
- сводные скаляры `k_F`, `k_E`, `k_pcr` с ДИ, отношения `pu/pcr`, `su/scr` и модуль разгрузки не формируются автоматически: для них нужны подтверждённые `pu`, `su`, `scr`, направление отношения и единая проектная методика;
- functional/mixed models, DOE, ANOVA/ANCOVA и поверхности отклика не запускаются автоматически: необходимы явно заданные факторы, блоки, структура парности и план эксперимента.
- нелинейная `custom_calibration` не угадывается: для неё потребуется отдельный версионированный контракт функции/таблицы; линейные возрастающие/убывающие и wrapped-шкалы уже поддержаны;
- HTML/XLSX-отчёт, полный проектный мастер, Windows EXE и расширенные циклические расчёты остаются последующими gated-итерациями; базовый CI для чистых Windows/Linux runner'ов добавлен отдельно.
- JSON ручного черновика является временным переносимым форматом TASK 12; для него поддержана последовательная lossless-миграция `manual-entry-draft/1.0` → `1.1` → `1.2`, причём общий паспорт остаётся read-only до инженерного поканального назначения. SQLite-архив, утверждение, ревизии, поиск, backup/restore и архивные миграции относятся к TASK 13–15 и здесь не имитируются;
- старые CSV/XLSX и metadata не требуют миграции и продолжают обрабатываться прежним pipeline; при откате на `0.4.1a2` новый JSON следует сохранить до возврата на ветку ручного ввода, поскольку старый GUI его не открывает.

Экспортируемые таблицы и пакет воспроизводимости позволяют добавить эти анализы без изменения исходного raw-протокола.
